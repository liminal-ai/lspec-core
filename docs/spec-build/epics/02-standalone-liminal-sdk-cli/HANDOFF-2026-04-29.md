## Project context

### What this package is

`lbuild-impl` is the **implementation runtime for liminal-build**. Liminal-build is an agentic workflow platform; this package provides the SDK and CLI primitives that drive the implementation phase of a spec pack — dispatching story implementors, running verification passes, applying bounded fixes, and recording durable artifacts.

This is *not* "lspec" or "lspec runtime." Lspec is the spec methodology (the format and skills used to *create* spec packs). This package *consumes* spec packs that lspec produces. Earlier in the project the package was named `@lspec/core` which conflated the two; we renamed.

### The structural issue I never raised

When Story 4 introduced the test-project split (`default` vs `integration` in `vitest.config.ts`), the design decision was "default = mocks only; integration = real provider CLIs, env-gated." The build-spawn tests and pack-install tests ended up in the `default` project by default — nobody, including me, looked at the categorization and asked "are these actually unit tests?"

**They are not.** They're integration tests of the build/pack/install pipeline. They:
- Run `npm run build` (full clean rebuild)
- Run `npm pack`
- Install the resulting tarball into a sandbox
- Spawn the built CLI binary
- Or run a TypeScript consumer against the installed package

That entire chain is platform-sensitive and slow. Putting these in the "fast unit" project means every PR build runs them, every push runs them, and the publish workflow's Layer A runs them. Each run, on Linux GHA, is fragile.

### Tests that probably belong in a different tier

Suspect set (search basis: tests that run `npm build`, `npm pack`, `tar`, or spawn the built binary):

- `tests/build-output.test.ts` — TC-1.4b (build + spawn `--help`)
- `tests/command/help.test.ts` — TC-3.1a (build + spawn `--help`)
- `tests/command/invocation.test.ts` — TC-3.5a (build + spawn through node)
- `tests/command/pack-and-install-smoke.test.ts` — TC-3.5b (build + pack + install + npx)
- `tests/sdk/operations.test.ts` — **TC-2.4b only** (build + node script importing the package)
- `tests/dist/metadata.test.ts` — TC-6.1a (reads package.json; probably fast, no spawn)
- `tests/dist/pack-install.test.ts` — TC-6.2a, TC-6.2b (build + pack + install + run)
- `tests/dist/types.test.ts` — TC-6.3a (build + pack + install + tsc)
- `tests/dist/subpath-imports.test.ts` — 3 tests (build + pack + install + run consumer)
- `tests/dist/version-sync.test.ts` — TC-6.4a (probably fast, no spawn)
- `tests/gorilla/distribution.test.ts` — TC-5.4b (`npm pack` + `tar`; not a build, but slow)
- `tests/foundation.test.ts` — TC-1.4a (build + existsSync; build is the slow part)

### The proposed three-tier split

Discuss with the user before implementing. This is a recommendation, not a decision.

**Tier 1: `default` project — fast in-process unit/contract tests**
- Schema validation, parser tests, code-quality tests, SDK contract tests, mock-provider tests
- Default vitest 5s timeout fine
- Runs on every push + PR via `ci.yml`
- Runs as Layer A of publish gate

**Tier 2: `package-shape` project (NEW) — package distribution + bin behavior**
- All build-spawn + pack-install tests listed above
- Slower (60-120s per test)
- NOT on every push/PR
- Triggered: workflow_dispatch, scheduled cadence (weekly?), or as part of publish gate
- Costs: ~2-5 min total runtime on Linux

**Tier 3: `integration` project — real-provider-CLI tests**
- `tests/integration/*.test.ts` (already exists; env-gated by `LSPEC_INTEGRATION=1`)
- Requires API keys + provider CLIs installed
- Currently runs Monday cron + tag publish

### Why this matters for the publish

If we keep the current tier mix, every retry of the publish workflow runs build-spawn tests on a Linux runner. Even with timeouts bumped to 120s, the failure surface is large because each test's timing varies. Fragile.

If we move them to a `package-shape` project that publish runs separately, the default-CI layer gets back to fast + reliable, and the package-shape layer is where slow-pipeline failures are expected and tolerated.

## Don't repeat my mistakes

### 1. Don't whack-a-mole CI failures

Each round, I fixed the visible failures and pushed. By round 3 the pattern was obvious — same root cause, different tests. The right move at round 1 was to ask "what category of tests is at risk on this platform?" and audit the whole category.

When you see a test failure, pause and ask:
- Is this test in the right project/tier for what it actually does?
- Is there a *category* of tests with the same property that will also fail?
- Should this be fixed at the test level, the categorization level, or the workflow level?

### 2. Don't make structural decisions silently

The test-project split decision (which tests live in `default` vs `integration`) was made implicitly by the placement of test files. Nobody — author, verifier, orchestrator — questioned it. When I encountered failures that hinted at the categorization being wrong, I didn't raise it; I bumped timeouts and moved on. The user had to stop me explicitly.

When you make a non-obvious decision, raise it. The user can disagree and you adjust. The user cannot disagree with a decision they didn't see.

### 3. Don't go from churn to autopilot

Earlier in the day I cycled through too many decision questions on naming and got pushed back as "churning." I overcorrected by going silent and just doing things. The user pushed back: "did you just go from churning to deciding everything and starting without a single confirmation?"

The middle is: confirm the consequential decisions, don't churn on the trivial ones, and explicitly flag your reasoning so the user can see the calibration choice.

### 4. Don't conditional-skip tests

User explicit preference: "conditional skips are bad form. I prefer fixing or deleting." A test that silently no-ops on one platform hides signal degradation. Either fix it to work everywhere or delete it; don't leave a `if (process.env.CI) return;` in the file.

### 5. Don't recommend "skip" 9 times in a row

Earlier today I compiled the cleanup batch and recommended "skip / accept-as-is" on every single one of 9 items. User immediately spotted the pattern as orchestrator bias (I had a stake in ratifying my own mid-cycle calls). When you find yourself recommending the same disposition for many items in a row, that's a smell — your reasoning may be biased toward the recommendation, not derived from the items.

### 6. Don't propose org/scope changes

Done deal. Unscoped `lbuild-impl` under `liminalbuilder` user account. Praxen later. Don't relitigate.
